# Yellow Club Mini App — MVP Release Checklist
**Cache version: v7.0.2**

---

## 1. Режимы запуска

### Реальный тест (сотрудники видят только свои роли)
```
MVP_RELEASE_MODE=true
WEB_APP_TEST_ROLES=false
```
- Тестовая панель ролей **полностью скрыта** (`canUseTestRoles=false` → `renderTestRolePanel()` добавляет `hidden`).
- Каждый сотрудник видит интерфейс только своей роли из базы.

### Внутренний тест (можно переключать роли)
```
MVP_RELEASE_MODE=true
WEB_APP_TEST_ROLES=true
```
- Тестовая панель ролей видна только `owner`/`operations`/пользователям из `ADMIN_IDS`.
- Dropdown содержит только 4 роли: Стажёр, Преподаватель, Методист, Админ (`client_manager` отфильтрован и на frontend, и на backend).
- Бейдж в шапке показывает `«Роль · тест»`.

> **Продакшн:** `MVP_RELEASE_MODE=true`, `WEB_APP_TEST_ROLES=false`

## 5. Чек-лист стажёра

1. Открыть приложение → видны только вкладки: **Стажировка · Помощь · Чат**.
2. Вкладка **Помощь** → заголовок «Инструкция стажёра», первый аккордеон «Быстрый маршрут» открыт.
3. Вкладка **Чат** → 7 чипсов стажёра видны под полем ввода.
4. Нажать чип «Что делать дальше?» → вопрос отправлен, агент отвечает с учётом текущего шага.
5. Вкладка **Стажировка** → отображается текущий шаг, кнопки соответствуют статусу.
6. Шаг 1 (наблюдение): можно записаться на занятие и написать комментарий.
7. Шаг 2 (работа): кнопка «Открыть материал» доступна, файл можно прикрепить.
8. После отправки файла → статус «На проверке».
9. При отклонении работы → виден комментарий методиста, кнопка повтора активна.
10. Нет вкладок Занятия / Задачи / Админ.

---

## 6. Чек-лист преподавателя

1. Открыть приложение → вкладки: **Занятия · Задачи · Помощь · Чат**.
2. Вкладка **Помощь** → заголовок «Инструкция преподавателя».
3. Вкладка **Чат** → 8 чипсов преподавателя (Один ученик, МойКласс, Подготовка и т.д.).
4. Вкладка **Занятия** → список занятий за 7 дней, статусы корректны.
5. Карточка занятия: заголовок, бейдж, дата/время; секции «Подготовка» и «Закрытие» доступны.
6. Загрузить файл подготовки → статус «На проверке».
7. После принятия работы администратором → статус обновился, зелёный индикатор.
8. Закрыть занятие: кнопка «Закрыть занятие» работает, статус меняется на «Закрыто».
9. Вкладка **Задачи** → карточки отражают актуальные занятия и статусы.
10. Нет вкладки Админ.

---

## 7. Чек-лист методиста / администратора

1. Открыть приложение → вкладки: **Занятия · Задачи · Помощь · Чат · Админ**.
2. Вкладка **Помощь** → заголовок «Инструкция методиста / администратора».
3. Вкладка **Чат** → 8 чипсов админа (Что требует проверки? и т.д.).
4. Вкладка **Админ** → видны подвкладки: Стажёры, Проверка работ, Контроль занятий, Преподаватели, Сотрудники, Уведомления.
5. **Стажёры**: карточки стажёров, статусы наблюдений и работ видны.
6. Принять / отклонить работу стажёра: стажёр видит обновление немедленно.
7. Допуск / недопуск после пробного: кнопки в карточке стажёра работают.
8. **Проверка работ**: список подготовок преподавателей, кнопки «Принять» / «Отклонить».
9. **Контроль занятий**: незакрытые занятия и проблемные видны.
10. Чат: «Что требует проверки?» → агент перечисляет ожидающие задачи.

---

## 8. Чек-лист чата (все роли)

1. Поле ввода открывается без зума (font-size ≥ 16px).
2. Чипсы соответствуют роли (см. таблицу выше).
3. Нажать любой чип → вопрос передаётся в чат, ответ приходит.
4. Переключение роли (в тестовом режиме) → чипсы и подсказки обновляются автоматически.
5. История чата сохраняется в рамках сессии.
6. Агент учитывает рабочий контекст (занятия, задачи, маршрут стажёра).
7. Вопросы на темы KB (Notion) получают ответ из базы знаний.

---

## 9. Что делать, если нашли баг

1. Зафиксировать: роль, вкладка, действие, ожидаемый результат, фактический результат.
2. Если ошибка в интерфейсе (текст, элементы) — скриншот и описание.
3. Если ошибка в ответе агента — скопировать вопрос и ответ полностью.
4. Если приложение зависло / белый экран — открыть консоль браузера, скопировать ошибки.
5. Передать разработчику: описание + роль + версия (v6.7.3).

## Food module (backend foundation, UI не включён)

**Статус:** backend foundation добавлен за флагом `FOOD_MODULE_ENABLED=false`.  
Staff MVP (intern/teacher/admin) не затронут. UI не менялся. Cache не бампался.

### Env flags
```
FOOD_MODULE_ENABLED=false              # по умолчанию. true — включить модуль питания
CAMP_LESSON_NAME_FILTER=Yellow Summer Week  # основной фильтр (совпадение в topic/name/groupName)
CAMP_LESSON_ALT_FILTERS=Summer Camp    # дополнительные фильтры через запятую (не YC1/YC2 — слишком широко)
```
> ВАЖНО: не добавляй YC1/YC2 в CAMP_LESSON_ALT_FILTERS — они совпадают по _prettyRoomName и дают ложные срабатывания для 45+ несвязанных занятий. Правильные занятия лагеря содержат "Yellow Summer Week" в topic/name.

### Что добавлено
- 6 новых SQLite таблиц: `camp_children`, `parent_child_links`, `food_menus`, `food_items`, `food_orders`, `food_order_items`
- 6 storage-методов: upsert_camp_child, list_camp_children, get_camp_child_by_mk_student_id, generate_child_link_code, get_child_by_link_code, link_parent_to_child
- `me()` теперь возвращает `foodModuleEnabled: bool`
- `POST /api/food/debug/sync-camp-children` — admin-only diagnostic endpoint

### Как протестировать diagnostic endpoint
```bash
# В .env: FOOD_MODULE_ENABLED=true
# Затем (с initData реального admin-пользователя):
POST /api/food/debug/sync-camp-children
{"save": false}   # диагностика без сохранения
{"save": true}    # диагностика + сохранение в camp_children
```

Подробная документация: `FOOD_MODULE_PLAN.md`

## Production deploy · Food module

### Шаги выкладки

1. Сделать backup проекта и БД.
2. Запушить код на сервер.
3. Проверить `.env`:
   ```
   FOOD_MODULE_ENABLED=true
   CAMP_ACTIVE_WEEK_MODE=auto
   FOOD_MENU_OCR_ENABLED=true   # или false — по необходимости
   ```
4. Dry-run — проверить, что будет удалено:
   ```
   python scripts/reset_food_module_for_production.py --dry-run
   ```
5. Проверить список таблиц и количество строк в выводе.
6. Выполнить реальный reset:
   ```
   python scripts/reset_food_module_for_production.py --confirm
   ```
   Backup создаётся автоматически в `backups/food_reset_YYYYMMDD_HHMMSS.sqlite`.
7. Запустить сервер.
8. Открыть Telegram Mini App, убедиться, что приложение стартует.
9. Зайти в Админ → **Питание · диагностика**:
   - Проверить блок «Состояние food data» — все счётчики должны быть 0.
   - Запустить диагностику (чекбокс «Сохранить найденных детей в БД»).
   - Обновить «Состояние food data» — `campChildren > 0`.
10. Зайти в **Питание · дети** — нажать «Сгенерировать коды» всем детям.
11. Зайти в **Питание · меню** — создать меню на текущий день, опубликовать.
12. Проверить родителя без привязки: код привязки работает.
13. Проверить, что старых заказов нет: **Питание · отчёт** → счётчик заказов = 0.
14. Родитель делает первый заказ — убедиться, что он отображается в сводке.
